Java 4 Features

Java 5 Features

  1. To improve the start up time of smaller applications, Class Data Sharing (CDS) was introduced. Anything which is loaded by the bootstrap classloader was serialized and stored in a file on disk that could be reloaded on future launches of the JVM. This meant that multiple instances of the JVM shared the class metadata, so it wouldn't have to load them all every time.

Java 8 Features

  1. Default Method in interface: A default method with body can be added to interfaces. The implementation needs to be provided in the interface itself. This method is automatically available to all the classes implementing the interface
  2. Static Method in interface: A method with keyword static can be added to interfaces. The implementation needs to be provided in the interface itself. This method can not be overridden in child classes implementing the interface
  3. Functional Interfaces: These are the interfaces having one and only one abstract method.
  4. Lambda Expression: These are the expressions used to create a function not belonging to any class. These are majorly used to implement a functional interface or in functional programming using Java Streams API.
  5. Process Termination: Process class has two new methods
    1. isAlive()
    2. destroyForcibly()
  6. StampedLocks: A new type of read write lock which is better in performance than standard read write lock.
  7. Concurrent Adders: These are lock free Java constructs which are better in performance than Atomic variables and are thread safe.

Java 9 Features

  1. Java 9 is about the modularization of the JDK and is also known as Project Jigsaw.
  2. G1GC has been made the default garbage collector now. It has replaced the Concurrent Mark and Sweep garbage collector.
  3. JIT as of Java 9 was written completely in C++. It converts Java bytecode into machine code at runtime. A new experimental interface called the JVM Compiler Interface or JVMCI has been introduced. This interface makes it possible to rewrite the JIT Compiler in pure Java.
  4. JShell and REPL: Java 9 introduces JShell and a Read-Eval-Print Loop (REPL) for the Java Programming Language. REPL allows you to evaluate code snippets such as declarations, statements, expressions. You can test your code as you create it, and way before you are done with your whole project.

Java 10 Features

  1. Release frequency: Java will release new features in every 6 months.
  2. Local Variable Type Inference: var
    • This not a kept as reserved keyword to maintain backward compatibility.
    • This is used to turn verbose local assignments to relatively simple assignment. e.g. HashMap<String, String> success = new HashMap<>(); can now be written as var success = new HashMap<, String>();. At compile time, java will figure out the reference type based on the value type.
  3. An unmodifiable view collection is a collection that is unmodifiable and is also a view onto a backing collection. Examples of unmodifiable view collections are those returned by the Collections.unmodifiableCollection, Collections.unmodifiableList, and related methods.UnmodifiableList returns an unmodifiable view collection. Note that changes to the backing collection might still be possible, and if they occur, they are visible through the unmodifiable view.
  4. List.copyOf(Collection collection) method is introduced where in collection becomes completely unmodifiable now
  5. 3 new methods have been introduced in Collectors class of Stream package
    1. toUnmodifiableList
    2. toUnmodifiableSet
    3. toUnmodifiableMap
  6. Performance enhancements to the default G1GC. Now full parallel processing happens.
  7. Application Class-Data Sharing:
    1. From Java 5 onwards, class metadata loaded by only bootstrap classloader was cached. Now the metadata loaded by system classloader and the platform classloader can also be cached. This can be done by adding flag -XX:+UseAppCDS
    2. Also the application classes can also be cached now. This can be done in 3 steps as below:
      1. Create list of classes to be cached
        java -Xshare:off -XX:+UseAppCDS -XX:DumpLoadedClassList=myapp.lst \ -cp $CLASSPATH $MAIN_CLASS
      2. Create the cache
        java -Xshare:dump -XX:+UseAppCDS -XX:SharedClassListFile=myapp.lst \ -XX:SharedArchiveFile=myapp.jsa \ -cp $CLASSPATH
      3. Run the app using the cache
        java -Xshare:on -XX:+UseAppCDS -XX:SharedArchiveFile=myapp.jsa \ -cp $CLASSPATH $MAIN_CLASS
  8. GRAAL: The JVM Compiler Interface or JVMCI introduced in JAVA 9 has been used to create complete Java based JIT compiler called GRAAL in Java 10. This JIT compiler as of now can be used in Linux/x64 machines only.
  9. Thread local handshakes: Till now GC process was stop-the-world process. In Java 10, the JVM can put an arbitrary number of threads into a safepoint, and threads may continue running after performing the prescribed "handshake". This resulted in the JVM being able to pause just one thread at a time, whereas before it had to pause them all.
  10. Numbering system: A new version numbering system with semantic $FEATURE.$INTERIM.$UPDATE.$PATCH has been introduced where
    • FEATURE is the version of Java.
    • INTERIM is reserved in case in future if java wants to release new features faster than 6 months.
    • UPDATE starts with 1 and increments by 1 for every three months.
    • PATCH is used for any critical release happening before UDPATE release.
    e.g. Java 10.0.1
  11. REPL Performance:The startup time of the JShell REPL tool has been improved a lot. It is Read-Eval-Print-Loop for Java language.
  12. SSL Enhancements: OpenJDK till Java 9 was having empty keystore. Now from Java 10 onwards, OpenJDK is also bundled with keystore called cacerts which is a home for root certificates for Certificate Authorities that the JVM can use to perform SSL handshakes. This can be confirmed by calling keytool -cacerts -list
  13. Alternative Memory Allocation: User can now use alternative memory devices such as Non-Volatile DIMM to allocate heap memory. This can be done using flag -XX:AllocateHeapAt=
  14. Container Awareness: JVM now knows if it is running inside Docker container. This means the application now has accurate information about what the docker container allocates to memory, CPU, and other system resources.Earlier, the JVM queried the host operating system to get this information. This causes a problem when the docker container would actually like to advertise a different resource set.

Frequently Asked Questions


What is the difference between Collections.unmodifiableList and Collectors.toUnmodifiableList?


Collections.unmodifiableList returns an unmodifiable view where in in case the backing collection is modified then the view will also get modified. But Collectors.toUnmodifiableList will return a completely unmodifiable list.
