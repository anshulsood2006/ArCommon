Beans in Springs

  • The beans are the objects that are instantiated, assembled and managed by spring's IOC container.
  • They are configured inside beans definition metadata which include below properties
    • id It is used to set unique name to identify the bean
    • name It is used to set unique name to identify the bean
    • class It is used to set class whose objects need to be created as beans
    • dependency-check It has been deprecated now. However before spring 3, it is used to check if the required properties have been set. It can have below values:
      • none No dependency check will take place.
      • simple Dependency check for only primitive and collection types. UnsatisfiedDependencyException will be thrown if the check fails.
      • objects Dependency check for only object types. UnsatisfiedDependencyException will be thrown if the check fails.
      • all Dependency check for all viz. primitive, collection and object types. UnsatisfiedDependencyException will be thrown if the check fails.
      After spring 3, the same thing can now be done using an annotation @Required on the setter method of the variable which is mandatory. Also below bean need to be registered <bean class="org.springframework.beans.factory.annotation.RequiredAnnotationBeanPostProcessor"/> to make this annotation work.
    • init-method It is used to call a method just after initialization of the bean.
    • destroy-method It is used to call a method just after initialization of the bean.
    • depends-on It is used to make sure that the bean with given name is created before the creation of this bean.

Dependency Injection in Springs

  • A normal program like below 
    public class Vehicle {
   private Engine engine;
   public Vehicle() {
      this.engine = new Engine();
   }
}
    can be written using dependency injection or an inversion of control scenario in two different ways.
    • Constructor Injection
      • public class Vehicle {
   private Engine engine;
   public Vehicle(Engine engine) {
      this.engine = engine;
   }
}
      • The Engine will be implemented independently and will be provided to the Vehicle at the time of Vehicle instantiation. Here control has been removed from the Vehicle and kept in an XML configuration file and the dependency i.e. class Engine is being injected into the class Vehicle through a Class Constructor.
      • The configuration needs to be done in bean configuration file is 
        <bean id="engine" class = "com.projects.Engine">
<bean id = "vehicle" class = "com.projects.Vehicle">
  	<constructor-arg ref = "engine"/>
</bean>
    • Setter Injection
      • public class Vehicle {
   private Engine engine;
   public void setEngine(Engine engine) {
      this.engine = engine;
   }
}
      • The Engine will be implemented independently and will be injected into Vehicle using a setter method.
      • The configuration needs to be done in bean configuration file is 
        <bean id="engine" class = "com.projects.Engine">
<bean id = "vehicle" class = "com.projects.Vehicle">
  	<property name = "engine" ref = "engine"/>
</bean>
  • Thus in both the above methods, the flow of control has been "inverted" by Dependency Injection (DI).
